 Step 5: Entity Cross-Referencing
 
   -- Cross-reference extracted entities against known heroes and watchlist
   CREATE OR REPLACE VIEW KYC_SUPERHERO_DB.CURATED.ENTITY_CROSSREF AS
   SELECT
     k.file_name,
     k.extracted_data:applicant_name::VARCHAR AS applicant_name,
     h.HERO_ID,
     h.HERO_NAME,
     h.RISK_LEVEL AS known_risk_level,
     h.AFFILIATION,
     w.REASON AS watchlist_reason,
     w.RISK_CATEGORY AS watchlist_category,
     CASE
       WHEN w.ENTITY_NAME IS NOT NULL THEN TRUE
       ELSE FALSE
     END AS watchlist_match,
     CASE
       WHEN h.RISK_LEVEL IN ('HIGH', 'CRITICAL') THEN TRUE
       WHEN w.ENTITY_NAME IS NOT NULL THEN TRUE
       WHEN k.extracted_data:politically_exposed::VARCHAR ILIKE '%yes%' THEN TRUE
       ELSE FALSE
     END AS edd_required
   FROM KYC_SUPERHERO_DB.CURATED.EXTRACTED_KYC k
   LEFT JOIN KYC_SUPERHERO_DB.RAW.HEROES h
     ON LOWER(k.extracted_data:applicant_name::VARCHAR) ILIKE '%' || LOWER(h.REAL_NAME) || '%'
     OR LOWER(k.extracted_data:applicant_name::VARCHAR) ILIKE '%' || LOWER(h.HERO_NAME) || '%'
   LEFT JOIN KYC_SUPERHERO_DB.RAW.WATCHLIST w
     ON LOWER(k.extracted_data:applicant_name::VARCHAR) ILIKE '%' || LOWER(w.ENTITY_NAME) || '%'
     OR LOWER(k.extracted_data:associated_entities::VARCHAR) ILIKE '%' || LOWER(w.ENTITY_NAME) || '%';

 Maker/Checker Architecture — Checker Layer for Steps 4-7
 
 Context
 
 The existing plan has a Maker layer that extracts and processes data. The Checker layer adds independent AI-driven validation at each step, mirroring real-world KYC compliance where dual-review (4-eyes principle) is mandatory. Each checker uses a separate AI call with a different prompt perspective to catch errors, hallucinations, or incomplete extractions.
 
 This is highly relevant to CPR — they already operate maker/checker processes for regulatory compliance. Showing this can be automated with AI will resonate strongly.
 
 ────────────────────────────────────────
 
 Checker Architecture
 
   MAKER (Extract)                    CHECKER (Validate)
   ─────────────────                  ──────────────────
   Step 4: AI_COMPLETE → ID data  →   Step 4C: Re-examine image, compare to maker output
   Step 5: AI_EXTRACT → KYC data  →   Step 5C: Cross-validate fields, check completeness
   Step 6: SQL JOINs → Entity XREF →  Step 6C: AI validates match quality, flags false +/-
   Step 7: AI EDD assessment      →   Step 7C: Independent AI review of EDD decision
 
 Each checker produces:
 
 • check_status: PASS / FAIL / NEEDS_REVIEW
 • confidence_score: 0-100
 • discrepancies: array of issues found
 • checker_notes: free-text reasoning

 Step 8: Final Maker/Checker Dashboard
 
   -- Combined dashboard showing maker + checker results with discrepancy flags
   CREATE OR REPLACE VIEW KYC_SUPERHERO_DB.ANALYTICS.MAKER_CHECKER_DASHBOARD AS
   SELECT
     'ID Extraction' AS stage,
     c.file_name,
     c.checker_output:check_status::VARCHAR AS check_status,
     c.checker_output:confidence_score::INT AS confidence,
     c.checker_output:document_authenticity::VARCHAR AS doc_authenticity,
     ARRAY_SIZE(c.checker_output:discrepancies) AS discrepancy_count,
     c.checker_output:checker_notes::VARCHAR AS notes
   FROM KYC_SUPERHERO_DB.CURATED.CHECKED_IDS c
   
   UNION ALL
   
   SELECT
     'KYC Form' AS stage,
     c.file_name,
     c.checker_output:check_status::VARCHAR AS check_status,
     c.checker_output:confidence_score::INT AS confidence,
     NULL AS doc_authenticity,
     ARRAY_SIZE(c.checker_output:discrepancies) AS discrepancy_count,
     c.checker_output:checker_notes::VARCHAR AS notes
   FROM KYC_SUPERHERO_DB.CURATED.CHECKED_KYC c;
   
   -- Summary statistics
   SELECT
     stage,
     COUNT(*) AS total_documents,
     SUM(CASE WHEN check_status = 'PASS' THEN 1 ELSE 0 END) AS passed,
     SUM(CASE WHEN check_status = 'FAIL' THEN 1 ELSE 0 END) AS failed,
     SUM(CASE WHEN check_status = 'NEEDS_REVIEW' THEN 1 ELSE 0 END) AS needs_review,
     AVG(confidence) AS avg_confidence,
     SUM(discrepancy_count) AS total_discrepancies
   FROM KYC_SUPERHERO_DB.ANALYTICS.MAKER_CHECKER_DASHBOARD
   GROUP BY stage;

 Key Design Principles
 
 1. Different perspective: Each checker uses a different "persona" and prompt angle from the maker — not just repeating the same extraction
 2. Structured output schemas: Every checker returns a consistent JSON structure enabling automated escalation
 3. Graduated severity: PASS/FAIL/NEEDS_REVIEW (or APPROVED/REJECTED/ESCALATE for EDD) maps to real operational workflows
 4. Audit trail: Every check is timestamped and stored, supporting regulatory requirements
 5. Score variance detection: Step 7C compares maker vs checker risk scores — large variance triggers escalation
